# 智能文档分析 Agent - 技术架构文档

## 1. 系统架构概览

本系统采用分层架构设计，通过 Agent 技术实现智能化的文档分析能力。

```
┌─────────────────────────────────────────────────────────┐
│                    用户交互层                          │
│  (Web UI / API / CLI)                                 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                   Agent 协调层                         │
│  (LangChain AgentExecutor)                             │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                   任务执行层                           │
│  (DocumentAnalyzer / InformationExtractor / ...)       │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                   文档解析层                           │
│  (PDFParser / DocxParser / ExcelParser / TxtParser)   │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                    LLM 接口层                         │
│  (OpenAI API / 本地模型 / MiMo 等)                    │
└─────────────────────────────────────────────────────────┘
```

## 2. 核心组件设计

### 2.1 DocumentAnalysisAgent（核心 Agent 类）

**职责**：
- 协调各个组件完成文档分析任务
- 管理对话历史和上下文
- 调用 LLM 进行语义理解

**关键方法**：
- `__init__()`: 初始化 LLM 和 Agent
- `_create_agent()`: 创建 LangChain Agent
- `analyze()`: 分析单个文档
- `batch_analyze()`: 批量分析多个文档

### 2.2 DocumentParser（文档解析器）

**设计模式**：策略模式（Strategy Pattern）

**支持的格式**：
- PDF（使用 PyPDF2）
- DOCX（使用 python-docx）
- XLSX/XLS（使用 openpyxl）
- TXT/MD（原生文件读取）

**扩展性**：可通过添加新的 Parser 类支持更多格式

### 2.3 API 服务层（FastAPI）

**提供的接口**：
- `POST /analyze`: 分析单个文档
- `POST /analyze/upload`: 上传并分析文档
- `POST /batch-analyze`: 批量分析文档
- `GET /supported-formats`: 获取支持的格式

## 3. 核心逻辑流详解

### 3.1 单文档分析流程

```
用户输入 → 文档路径 + 任务描述
    ↓
[1. 文档解析阶段]
    - 识别文档格式
    - 调用对应 Parser
    - 提取文本内容
    ↓
[2. 内容理解阶段]
    - 构建 Prompt
    - 调用 LLM 进行语义理解
    - 提取关键信息
    ↓
[3. 任务执行阶段]
    - 根据任务类型执行相应逻辑
    - 总结 / 提取 / 问答 / 分类
    ↓
[4. 结果生成阶段]
    - 格式化输出
    - 返回结构化结果
```

### 3.2 Agent 决策流程

```
用户输入
    ↓
Agent 接收任务
    ↓
[思考] 需要什么工具？
    ↓
[行动] 调用相应工具
    ↓
[观察] 获取工具返回结果
    ↓
[思考] 任务是否完成？
    ↓
是 → 返回最终结果
否 → 继续思考和行动
```

## 4. 技术选型说明

### 4.1 为什么选择 LangChain？

- **Agent 框架成熟**：提供了完整的 Agent 构建工具
- **工具集成简单**：可以方便地添加自定义工具
- **记忆管理**：内置多种记忆管理机制
- **可扩展性强**：支持自定义 Agent 和工具

### 4.2 为什么选择 FastAPI？

- **高性能**：基于 Starlette 和 Pydantic
- **自动文档**：自动生成 OpenAPI 文档
- **数据验证**：使用 Pydantic 进行数据验证
- **异步支持**：原生支持 async/await

### 4.3 为什么使用策略模式解析文档？

- **开闭原则**：对扩展开放，对修改关闭
- **单一职责**：每个 Parser 只负责一种格式
- **易于测试**：可以单独测试每个 Parser
- **灵活性高**：运行时动态选择 Parser

## 5. 扩展性设计

### 5.1 支持新的文档格式

只需三步：
1. 创建新的 Parser 类（继承 BaseParser）
2. 实现 `parse()` 方法
3. 在 DocumentParser 中注册

### 5.2 支持新的分析任务

只需两步：
1. 在 Prompt 中描述新任务
2. Agent 会自动理解并执行

### 5.3 支持新的 LLM 后端

只需修改配置：
- 修改 `.env` 中的 `OPENAI_BASE_URL`
- 可以接入任何兼容 OpenAI API 的模型（包括 MiMo）

## 6. 性能优化考虑

### 6.1 大文档处理

- **分块处理**：将大文档分成多个块
- **并行处理**：使用多线程/协程并行处理多个块
- **增量分析**：只处理新增或修改后的内容

### 6.2 缓存机制

- **文档内容缓存**：避免重复解析同一文档
- **LLM 响应缓存**：缓存相同问题的回答
- **向量索引缓存**：缓存向量数据库的索引

## 7. 安全性设计

### 7.1 输入验证

- 验证文件格式
- 验证文件大小
- 防止路径遍历攻击

### 7.2 API 安全

- 支持 API Key 认证
- 限制请求频率
- 输入内容过滤

### 7.3 数据安全

- 不永久存储用户文档
- 临时文件及时清理
- 支持本地部署，数据不出域
